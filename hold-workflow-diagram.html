🟡 Hold Feature — How Re-attempts Work
1️⃣
Supervisor submits Lot Avg
Data saved as stages['lot_avg'] with pending status. Manager sees this in their Approvals page.
2️⃣
Manager puts it on HOLD
Status changes to hold. A snapshot is saved in holdHistory['lot_avg']. The supervisor is notified — they need to re-submit with corrected data.
3️⃣
Supervisor re-submits (Attempt 2)
✅ NEW BEHAVIOR: Instead of overwriting the original, the system creates a separate key — stages['lot_avg_reattempt_2'] — with pending status. The original stages['lot_avg'] stays as hold.
(All data preserved — nothing is lost!)
4️⃣
Repeat up to 4 times
Each re-submission creates a new key:
lot_avg_reattempt_2 → lot_avg_reattempt_3 → lot_avg_reattempt_4
All 4 attempts preserved with their own data and status. Max 4 attempts per stage.
